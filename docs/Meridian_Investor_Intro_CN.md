# Meridian — 面向投资人的项目介绍

## 一句话定位

Meridian 是**多模型、多渠道的 AI 编程调度中枢**：用户通过统一入口（如 Telegram）选择模型、下发任务，由唯一控制平面路由到不同 CLI/Agent 或 A2A 服务并回传结果——相当于为「谁来做、从哪进、到哪去」提供可扩展的协议与实现。

## 市场价值

- **AI 编程与多模型并存的刚需**：开发者与团队会同时使用 Claude、Codex、Gemini、Cursor 以及国内大模型；需要一个**统一控制平面**来管理会话、实例与任务分发，而不是每个模型各搞一套入口与运维。
- **协议即护城河**：基于 A2A（Agent-to-Agent）标准与自研 Hub 协议，任何实现「能力声明 + 任务接收 + 结果回传」的服务都可接入，无需改 Hub 核心。调度层与具体模型/后端解耦，便于**多厂商、多形态**的 Agent 与自动化服务接入。
- **渠道与后端双扩展**：入口侧支持 Telegram，可扩展到 Email、Nostr、WhatsApp 等；执行侧支持 agentapi（Claude/Codex/Gemini/Cursor）与任意 A2A Service（如 CORD）。同一套架构可支撑**从个人到企业、从单机到多实例**的部署形态。

## 中国自己的模型，可以有属于自己的「类 OpenClaw」生态

OpenClaw 代表了一类生态：**以即时通讯（如 Telegram）为入口，统一展示多模型、多 Provider，用户选模型、发任务，由网关与调度层把请求交给对应 CLI/API，并回收结果**。这一套体验今天严重依赖海外模型与海外闭环。

Meridian 的架构与协议表明：**中国自己的大模型（通义、文心、智谱、月之暗面、DeepSeek 等）完全可以复刻并超越这一生态**：

- **同一套 Hub 与协议**：调度中枢与 A2A 协议与具体模型无关。国内厂商或集成方只需实现「Agent Card + tasks/send + HubResult」的 Service，即可挂到 Meridian 上，与现有 Claude/Codex 等并列出现，用户在同一入口切换国内外模型。
- **数据与合规可控**：请求与结果经自有 Hub 与自建 Service 闭环，可全部落在国内 infra，满足数据本地化与审计要求；模型侧可完全选用国产 API 与私有化部署。
- **生态可自有、可商用**：Telegram/多端作为前端入口，Meridian 作为调度与协议层，后端接国内模型与自研 Agent（如 CORD），即构成「中国自己的类 OpenClaw」：同一体验、自主可控、可面向 To B/To C 做产品化与商业化。

## 发展愿景：从调度中枢到「内部 AI 系统」统一架构

Meridian 的演进方向，是成为**企业 / 个人内部的 AI 系统统一架构**（可对标内部架构文档中的「v3」形态）：不是单一产品，而是一套可落地、可扩展的分层范式。

- **入口层（Interface）**：从单一 Telegram 扩展到多通道——Email、Nostr、WhatsApp、企业 IM、Web 控制台、API。所有入口归一为统一事件（如 InboundUIEvent / HubMessage），由同一控制平面处理。
- **控制平面（Hub Core）**：唯一调度与编排层。路由、实例生命周期、可观测（trace_id、日志、审计）集中在此；支持内置 Agent（agentapi）与外部 A2A Service 的混合编排，未来可扩展策略、配额、多租户。
- **执行层（Agents / Services）**：多模型 CLI、CORD 类研发闭环、企业自研 Agent、第三方 A2A 服务，均通过协议注册与发现，按 intent 调度。**愿景**是：任何「能声明能力、能收任务、能回结果」的组件，都可即插即用，形成企业内部的「AI 服务网格」。
- **可观测与治理**：全链路 trace、日志、Monitor 状态与事件，为合规、审计、成本分摊和体验优化提供统一底座。

这一愿景下的 Meridian，本质是**内部 AI 系统的中控架构**：一个 Hub、多通道入口、多 Agent/Service 后端，协议开放、数据与调度可控，既可服务于个人/小团队，也可平滑扩展到企业级部署。

## 传统企业 AI 化：Meridian 扮演的关键角色

传统企业的 AI 化，痛点往往不是「没有模型」，而是**入口碎片化、模型与任务无法统一编排、数据与合规难以落地**。Meridian 型架构恰好对应这三类需求：

- **统一入口，降低采纳门槛**：员工通过已有习惯的渠道（如企业 IM、邮件、内部门户）与 AI 交互，无需学习多套工具。企业只需对接一次 Meridian Hub，即可在背后灵活切换或扩容模型与 Agent，对用户保持单一体验。
- **编排与合规一体化**：所有请求与结果经 Hub 流转，天然具备审计轨迹、策略注入（如敏感任务必须走国产模型、指定项目走私有化部署）的能力。数据留在企业边界内，模型可选公有云 API 或私有化，满足数据本地化与行业监管要求。
- **从「上一个大模型」到「上一套 AI 体系」**：企业可逐步接入自研 Agent、采购的 A2A 服务、以及多厂商模型，由 Hub 按 intent/策略路由。Meridian 不绑定某一家模型或某一种执行形态，而是提供**企业 AI 化的控制平面与协议标准**，便于 IT 与业务在统一底座上迭代。

对传统企业而言，Meridian 的生态位是：**介于「人的意图」与「多种 AI 执行端点」之间的标准层**——谁占据这一层，谁就定义了企业内 AI 的入口形态、数据流向与可观测性，从而支撑从试点到规模化 AI 化。

## 一人公司 / 小团队趋势：AI 时代的「操作系统」级生态位

「一人公司」、微型团队、独立开发者正在成为常态：人少、事多、需要 AI 作为力量倍增器。但若每个能力都对应一个独立产品（写作一个、代码一个、运营一个），会形成**工具碎片化、上下文割裂、运维成本高**的问题。

Meridian 在这一趋势下可定位为**个人/小团队的「AI 操作系统」**：

- **一个入口，多模型多能力**：通过同一入口（如 Telegram 或后续 Web）选模型、发任务、收结果。编码、写作、分析、自动化等能力以 Agent/Service 形式挂载到同一 Hub，用户只需与一个「控制台」交互，而不是在多个 Tab、多个账号之间切换。
- **协议开放，自建与托管并存**：个人可以只接公有模型 + 少量自建 Agent；小团队可以接入 CORD 等研发闭环 + 内部工具封装为 A2A Service。同一套协议与 Hub，从「一人」到「几人到几十人」可平滑扩展，无需重写架构。
- **可观测与可复现**：trace_id、日志、任务历史使个人或小团队也能做简单的复盘与审计（「上次这个需求是谁/哪个模型完成的？」），为知识沉淀和协作留出空间。

在一人公司与小团队场景中，Meridian 的生态位是：**介于「单一个体/小团队」与「多模型、多 Agent、多工具」之间的统一控制层**。不替代具体模型或具体应用，而是成为**串联所有 AI 能力的中枢**，让人把精力放在业务与产品上，而不是反复拼接工具链。

## 为什么值得投资

- **卡位「多模型调度与编排」**：在「多模型并存」成为常态的背景下，谁提供统一、可扩展的调度与协议层，谁就占据入口与集成的话语权；Meridian 已具备可运行的 Hub、Interface、Monitor 与 A2A 接入路径，可直接作为国内多模型生态的基础设施之一。
- **技术路径清晰、可验证**：Phase 0 已实现 Telegram → Hub → agentapi → 多 CLI；A2A 与 OpenClaw 侧 callMeridian 的联调路线已在协议与任务 spec 中明确，便于快速验证「国产模型 + 同一 Hub」的端到端体验。
- **可复用的生态叙事**：不绑定单一模型或单一渠道，故事可以讲成「为全球多模型与多 Agent 提供调度与协议」，在中国则具体化为「为国产模型打造自主可控的类 OpenClaw 生态」——同一技术底座，两种市场叙事，有利于融资与商务拓展。
- **双场景叠加**：同一套架构同时服务「传统企业 AI 化」（统一入口、编排、合规）与「一人公司 / 小团队」（AI 操作系统、一个入口多能力），两类客户共用协议与控制平面，边际扩展成本低、想象空间大。
